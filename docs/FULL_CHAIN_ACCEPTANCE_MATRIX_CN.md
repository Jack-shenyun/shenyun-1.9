# 企业管理系统全链路验收测试矩阵

## 1. 目标

本矩阵用于执行一轮接近真实运营的全链路业务验收，核心目标如下：

- 模拟真实用户跨部门协作
- 覆盖销售、审批、采购、生产、质量、实验室、仓库、财务、批记录、标签/UDI、文件归档
- 验证每一笔业务是否可以形成完整记录链
- 验证所有关键环节是否可追溯
- 提前发现上线后最容易卡住业务的真实问题

本轮验收重点不是“页面能不能打开”，而是：

- 单据是否能走通
- 状态是否能回写
- 审批是否命中正确节点
- 记录是否完整
- 文件是否生成
- 文件是否能保存并归档
- 批号、单号、金额、时间、责任人是否能串起来

## 2. 主链路

默认主链路如下：

1. 销售基础条件设置
2. 客户/产品/价格/付款条件准备
3. 报价或销售订单
4. 审批流
5. 自动分流到采购或生产
6. 采购计划/生产计划
7. 生产执行
8. 生产记录
9. 批记录汇总
10. 质量记录（IQC/IPQC/OQC）
11. 实验室记录
12. 入库申请
13. 仓库入库/出库
14. 标签打印 / UDI 生成与上传
15. 财务应收/应付/收付款
16. 文件生成
17. 文件归档到知识库
18. 全链路追溯回查

## 4. 20 组测试场景矩阵

### A. 销售与审批基础场景

#### Case 01 标准账期订单

- 目的：验证标准销售到审批的主链
- 条件：国内客户、账期支付、常规产品、无需报关
- 关键检查：
  - 销售订单创建
  - 审批节点是否命中预期或进入兜底
  - 审批记录是否完整
  - 是否自动生成应收
  - 是否自动生成生产/采购计划

#### Case 02 预付款订单

- 目的：验证预付款对后续生产触发的影响
- 条件：预付款客户、生产型产品
- 关键检查：
  - 审批通过后是否立即排产
  - 收款前是否拦截生产
  - 收款后是否自动触发生产计划

#### Case 03 先款后货订单

- 目的：验证先款后货条件控制
- 条件：先款后货、库存不足
- 关键检查：
  - 生产是否被提前放行
  - 财务收款后状态是否变化
  - 出货前是否仍校验收款条件

#### Case 04 货到付款订单

- 目的：验证货到付款分支
- 条件：货到付款、常规产品
- 关键检查：
  - 应收单生成逻辑
  - 发货与收款顺序是否允许
  - 后续财务记录是否匹配

#### Case 05 含运费/报关/附件订单

- 目的：验证销售附加条件
- 条件：需要运费、报关、上传附件
- 关键检查：
  - 运费字段是否影响金额与记录
  - 报关待办/报关记录是否生成
  - 附件是否可上传并归档

### B. 销售分流到采购/生产

#### Case 06 库存充足订单

- 目的：验证库存足够时不应误触发排产
- 条件：库存充足、生产型产品
- 关键检查：
  - 是否错误生成生产计划
  - 是否可直接进入待发货逻辑

#### Case 07 库存不足订单

- 目的：验证库存不足时自动排产
- 条件：库存不足、生产型产品
- 关键检查：
  - 是否生成生产计划
  - 计划数量是否等于缺口
  - 后续生产链是否能接上

#### Case 08 采购型产品订单

- 目的：验证采购型产品分流
- 条件：采购型产品、库存不足
- 关键检查：
  - 是否进入采购计划而非生产计划
  - 供应商信息是否自动带出
  - 后续 IQC 是否能接上

#### Case 09 混合订单

- 目的：验证同一订单多产品分流
- 条件：一部分采购型、一部分生产型
- 关键检查：
  - 是否分别生成采购与生产计划
  - 追溯时能否回到同一销售订单

#### Case 10 销售改单重算

- 目的：验证改单后的计划与记录一致性
- 条件：创建后修改数量、交期、付款条件
- 关键检查：
  - 原计划是否清理或更新
  - 应收是否同步
  - 审批是否重新进入正确流程

### C. 采购与来料质量

#### Case 11 标准采购到 IQC 合格

- 目的：验证采购主链
- 条件：采购计划 -> 采购订单 -> 来料 -> IQC 合格
- 关键检查：
  - 采购订单状态
  - 入库状态
  - IQC 记录
  - 合格后库存状态

#### Case 12 IQC 不合格

- 目的：验证不合格来料拦截
- 条件：采购到货 -> IQC 不合格
- 关键检查：
  - 是否阻止进入合格库存
  - 是否生成质量记录/异常记录
  - 后续采购/仓库状态是否错乱

### D. 生产执行与生产记录

#### Case 13 标准生产完成

- 目的：验证生产主链
- 条件：销售驱动生产、正常完工
- 关键检查：
  - 生产计划 -> 生产指令 -> 生产记录
  - 完工是否回写销售订单/计划
  - 批记录是否生成

#### Case 14 部分完工

- 目的：验证部分完成场景
- 条件：生产一部分完成
- 关键检查：
  - 计划状态
  - 生产记录
  - 批记录是否出现部分数据
  - 后续质量/仓库是否能部分处理

#### Case 15 返工/补录/重录

- 目的：验证生产记录鲁棒性
- 条件：修改生产记录、补录生产记录
- 关键检查：
  - 原记录是否保留
  - 批记录是否重复或断裂
  - 关联计划/指令是否紊乱

#### Case 16 灭菌链路

- 目的：验证灭菌后到质量
- 条件：需灭菌产品
- 关键检查：
  - 灭菌单状态
  - 到货后是否通知质量部 OQC
  - 批号与灭菌批号是否一致

### E. 质量、实验室、入库、仓库、财务

#### Case 17 IPQC 与实验室记录

- 目的：验证生产中质量与实验室数据
- 条件：过程检验、实验室记录录入
- 关键检查：
  - IPQC 是否与生产指令关联
  - 实验室记录是否进入追溯链
  - 电子签名/附件是否完整

#### Case 18 OQC 合格放行

- 目的：验证成品放行主链
- 条件：OQC 合格
- 关键检查：
  - OQC 记录
  - 是否回写生产入库申请
  - 仓库是否收到可入库信号

#### Case 19 OQC 不合格拦截

- 目的：验证成品不合格时的拦截
- 条件：OQC 不合格
- 关键检查：
  - 是否阻止入库/出库
  - 质量记录是否完整
  - 批记录是否能保留异常信息

#### Case 20 仓库到财务闭环

- 目的：验证最终业务闭环
- 条件：入库、出库、发货、应收、收款、文件归档
- 关键检查：
  - 仓库出库记录
  - 销售发货状态
  - 应收状态
  - 收款记录
  - 销售订单收款状态是否同步
  - 文件是否能归档知识库

## 5. 每组场景必须产出的记录

每组测试至少要求保留以下证据：

- 测试编号
- 测试时间
- 测试人/角色
- 输入条件
- 关键业务单号
- 关键批号
- 关键审批节点
- 关键截图或导出文件
- 结果结论
- 问题描述

## 6. 每组场景必须核对的文件

按场景不同，至少核对以下文件是否能生成：

- 销售订单打印件
- 发货单
- 采购单/入库单
- 生产记录打印件
- 检验记录/检验报告
- 标签打印文件
- UDI 输出文件
- 收款相关文件

并核对：

- 文件命名是否规范
- 文件内容是否带正确单号/批号/客户/产品
- 文件是否能被上传或归档

## 7. 每组场景必须核对的追溯点

从最终结果反查时，至少要能找到：

- 销售订单
- 审批记录
- 采购或生产计划
- 生产指令
- 批记录
- 质量记录
- 实验室记录
- 入出库记录
- 财务记录
- 文件记录

如果任意一层无法稳定反查，则该场景判定为“追溯不完整”。

## 8. 问题分级

### P0

- 会直接卡死业务
- 无法继续下一环节
- 会导致上线后核心流程不可用

### P1

- 主链能走，但闭环不完整
- 状态不同步
- 追溯断裂
- 记录缺失

### P2

- 功能可用但高风险
- 异常场景容易错
- 模板/文件/签名/上传链不稳定

### P3

- 细节不一致
- 展示问题
- 非核心流程缺陷

## 10. 当前已知高风险点

基于目前已经做过的真实测试，至少存在以下高风险点：

- 财务收款后，销售订单 paymentStatus 可能不回写
- 审批节点可能进入系统管理员兜底，而不是命中预期审批人
- 账期相关 dueDate 逻辑需要专项验证
- 生产、质量、仓库、财务虽然已有联动，但异常场景尚未逐组验证
