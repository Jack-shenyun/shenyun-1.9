## 整改清单

### P1：优先整改

1. 关键表单大量扩展字段仍塞在 `remark` 里保存 JSON。
原因：IPQC、OQC、领料单、生产入库申请等核心单据的结构化内容没有拆成正式字段或子表，后续查询、审计、导出、3Q 取证都很吃力。

2. 生产入库申请存在重复语义字段。
表现：`modifyReason` 和 `quantityModifyReason` 两个字段表达的是同一件事，容易出现填写不一致。

3. 审批层级偏多，不适合 15 人规模团队。
表现：采购单同时维护 `deptApprovalStatus`、`gmApprovalStatus`、`orderApprovals`；销售、生产也有独立审批记录，维护成本高。

4. 单据之间仍有“人工搬运信息”的影子。
表现：到货单到入库、OQC 到生产入库、销售订单到出库，虽然已有关联字段，但很多页面仍靠单号或批号去带数据，容易串错。

5. 删除留痕过多依赖回收箱，而不是状态禁用。
表现：部分业务对象更适合“作废/停用”，不是物理删除进回收箱。

### P2：建议尽快优化

1. 采购辅料和低风险物料不需要和关键原料走同样长的流程。
2. 销售预付款订单不必在付款前创建过多下游单据。
3. 委外灭菌完成后，系统应优先自动生成待 OQC / 待入库待办，而不是依赖人工补单。
4. 财务应收应付、发票、收付款记录之间应默认自动联动并自动回填状态。
5. 操作日志建议补“业务单号”统一字段，便于跨模块追溯。

### P3：整理性优化

1. 统一所有主流程单号命名规则。
2. 统一“审批通过 / 已审批 / 已确认 / 已完成”这些相近状态的边界。
3. 为小团队增加“简化模式”参数，不要所有公司都走同一套重流程。

## 过度设计的流程

### 1. 采购链路偏重

当前典型链路：

`采购申请 -> 采购订单 -> 部门审批 -> 总经理审批 -> 到货 -> IQC -> 入库 -> 应付 -> 收票 -> 付款`

对 15 人使用规模来说，建议改成两档：

1. 关键原料/医疗器械相关物料：保留全流程
2. 低风险辅料/行政类采购：简化成 `采购申请 -> 采购订单 -> 收货入库 -> 付款`

### 2. 生产链路偏重

当前典型链路：

`生产计划 -> 生产指令 -> 领料单 -> 生产记录 -> 工艺流转卡 -> 委外灭菌单 -> OQC -> 生产入库申请 -> 生产入库`

建议简化：

1. `工艺流转卡` 与 `委外灭菌单` 可做强关联自动生成
2. `OQC 合格` 后可自动生成 `生产入库申请`
3. 对不需要灭菌的产品，跳过灭菌相关节点

### 3. 销售链路状态偏多

当前状态粒度较细，适合大团队和复杂分工；对现阶段更实用的状态建议收敛为：

1. 草稿
2. 待审批
3. 待付款
4. 待发货
5. 已发货
6. 已完成
7. 已取消

### 4. 财务协同可再自动化

建议：

1. 销售发货后自动生成应收
2. 采购入库后自动生成应付
3. 发票默认关联应收/应付
4. 收付款后自动回写应收/应付状态

## 适合你们当前团队的简化方案

建议按“关键单据严格、普通单据轻量”的原则运行：

1. 销售订单保留审批，但常规国内订单只保留 1 级审批
2. 采购订单取消“部门 + 总经理”双审批，改成金额阈值审批
3. 低风险辅料免单独 IQC，改抽检模板
4. 生产中不再强制每单都建流转卡，只有灭菌产品建
5. OQC 合格后自动推送入库，不再手工重复录入
6. 收票、付款、收款只保留财务确认，不再额外走审批流
7. 删除动作优先改成“作废/停用”，减少回收箱依赖
