# 业务逻辑变更日志

> 用途：记录系统级业务逻辑调整，方便后续整体验证、回归测试、流程复盘。  
> 约定：只记录“逻辑变更”，不记录单纯样式调整或单条数据修正。

---

## 2026-03-16

### 1. 编码规则统一收口，不再允许前端或业务分支绕过系统编号

- 目的：
  - 避免“编码设置显示 0，但业务单号已经好几百”这种口径不一致。
  - 避免前端本地拼号、自动排产时间戳拼号、培训页面假编号等绕过正式规则的情况。
- 逻辑：
  - `生产计划` 自动排产统一改为调用系统编码规则，不再使用 `PP-日期-时间戳后四位`。
  - `培训记录` 编号正式纳入编码规则体系，编码设置读取规则时会先确保 `trainingNo` 列存在并完成回填。
  - 协同公司自动生成的内部销售单、内部采购入库单，不再自己拼 `SO-IC-*` / `IC-IN-*`，统一回到销售订单和入库单正式编号规则。
  - `编码设置` 页面遇到重复规则时，不再取第一条脏数据，而是优先显示真正有流水的规则。
  - `编码设置 -> 重新同步流水号` 现在会调用后端重新按真实业务单号扫描，而不是前端假重置。
- 影响文件：
  - `server/db.ts`
  - `server/routers.ts`
  - `client/src/pages/settings/Codes.tsx`

### 2. 培训管理不再本地假保存，正式走后端编号与落库

- 目的：
  - 避免培训页面看起来能新增，但实际上只是前端本地拼出 `TR-2026-001` 之类假编号，后端没有真正保存。
- 逻辑：
  - 新建培训时，编号改为“保存后系统生成”。
  - 保存时正式调用后端 `trainings.create / trainings.update`。
  - 培训状态统一改成后端真实枚举：`planned / in_progress / completed / cancelled`。
  - 培训类型统一改成系统真实枚举：`onboarding / skill / compliance / safety / other`。
- 影响文件：
  - `client/src/pages/admin/Training.tsx`

### 3. 产品修改后，引用单据自动同步

- 目的：
  - 避免只改了产品主数据，但采购计划、采购前置单据仍保留旧供应商/旧名称/旧单位，导致后续创建报错。
- 逻辑：
  - 当产品主数据被修改后，系统自动同步“未执行完成”的引用单据快照。
  - 当产品默认供应商被新增、修改、删除后，也会触发同样的同步。
- 当前同步范围：
  - 采购计划 `production_plans`
  - 未开始的生产指令单位快照 `production_orders`
  - 可编辑状态的物料申请明细 `material_request_items`
  - 草稿/审批中的采购订单明细 `purchase_order_items`
- 同步原则：
  - 只同步“前置/未完结单据”
  - 不回写已执行完成的历史单据，避免破坏追溯留痕
  - 默认供应商不存在时，旧引用中的供应商字段会自动清空
- 触发入口：
  - 产品管理修改产品
  - 产品管理修改默认供应商
  - 产品管理删除默认供应商
- 影响文件：
  - `server/db.ts`

- 逻辑：
  - 采购计划跳转时带 `returnTo=/purchase/plan`
  - 产品保存成功后，如果存在 `returnTo`，自动返回来源页面
  - 直接从产品管理自行编辑时，不影响原行为
- 影响文件：
  - `client/src/pages/purchase/PurchasePlanBoard.tsx`
  - `client/src/pages/rd/Products.tsx`

### 6. 采购订单提交审核统一走正式提交流程

- 目的：
  - 避免采购订单详情页点击“提交审核”时仅改状态、不走正式提交流程，且失败原因没有提示，用户误以为“提交无效”。
- 逻辑：
  - 采购订单详情页草稿状态下的“提交审核”统一调用 `submitForApproval` 接口。
  - 前端不再先行弹出成功提示，而是以后端提交成功为准。
  - 后端提交审核时会先读取采购订单本身，再把采购单号、供应商名称一起带入审批流，保证待办和审批记录来源一致。
  - 创建/更新采购订单失败时，前端会明确提示后端返回的校验原因。
- 影响范围：
  - 采购订单详情提交审核
  - 采购订单审批流发起
  - 采购订单失败提示
- 影响文件：
  - `client/src/pages/purchase/Orders.tsx`
  - `server/routers.ts`

### 7. BOM 保持完整，生产领料与生产记录自动带料单独过滤包装材料

- 目的：

- 逻辑：

  - MRP / 物料运算继续按完整 BOM 计算。
  - 生产领料建议单独过滤 `packaging_material` 和 `consumable`。
  - 生产记录按 BOM 自动带料时，同样过滤 `packaging_material` 和 `consumable`。
- 影响范围：
  - BOM 完整展示
  - MRP 运算
  - 生产领料建议
  - 生产记录自动带料
- 影响文件：
  - `shared/productCategories.ts`
  - `server/db.ts`
  - `client/src/pages/production/ProductionRecord.tsx`

### 8. “我处理的”统一按操作日志闭环，审批与收付款动作都写入处理日志

- 目的：
  - 避免“待办中心 -> 我处理的”只能看到部分模块，用户真实处理过的审批、付款、收款却不显示。
- 逻辑：
  - 所有配置审批流的单据，在“审批通过/驳回”时统一自动写入 `operation_logs`。
  - 所有通过底层 `createPaymentRecord` 创建的付款、收款记录，统一自动写入 `operation_logs`。
  - “我处理的”与顶部 `myProcessed` 统计继续只读 `operation_logs`，不再按模块分散特判。
  - 财务处理日志路由统一：
    - 收款 -> `/finance/receivable`
    - 付款 -> `/purchase/finance`
- 影响范围：
  - 待办中心 `我处理的`
  - 首页/右上角待办统计里的已处理计数
  - 审批通过/驳回、付款、收款等处理动作
- 影响文件：
  - `server/db.ts`

### 17. 采购订单下达后自动生成应付账款

- 目的：
  - 避免采购订单已经下达，但采购部「财务协同」页没有任何数据，导致采购付款链断开。
- 逻辑：
  - 采购订单状态进入 `issued / ordered / partial_received / received / completed` 时，系统自动确保存在一条应付账款。
  - 如果该采购订单已经有系统自动生成的应付账款，则自动同步供应商、金额、到期日等字段。
  - 如果该采购订单已经存在人工维护的应付账款，则不重复生成。
  - 采购订单编辑时也会复用同一套逻辑，不再依赖页面手工补应付。
- 影响范围：
  - 新建采购订单直接下达
  - 无审批模式下的采购订单提交
  - 审批通过后的采购订单下达
  - 已下达采购订单编辑后同步应付
- 影响文件：
  - `server/db.ts`
  - `server/routers.ts`

### 18. 采购自动应付继承供应商付款条件，采购财务协同状态改为“已提交财务”

- 目的：
  - 避免采购订单下达后自动生成的应付一律写成 `银行转账`，导致账期供应商被错误分到现款付款列表。
  - 避免采购部提交财务协同后，页面状态仍显示成财务视角的 `待付款`，和业务动作不一致。
- 逻辑：
  - 自动生成采购应付时，付款条件优先继承供应商资料中的 `paymentTerms`，再回退默认值。
  - 因此 `账期支付` 供应商生成的应付会自动进入账期/对账链，不再混进现款付款列表。
  - 在采购部 `财务协同` 页面里，状态 `pending` 统一显示为 `已提交财务`，表示采购已提交、等待财务执行。
- 影响范围：
  - 采购订单下达后自动生成应付
  - 采购部财务协同列表和详情状态文案
- 影响文件：
  - `server/db.ts`
  - `client/src/pages/finance/Payable.tsx`

  - 让“逻辑统一匹配”和“实际绑定对象展示”同时成立，便于后续整体验证运行逻辑。

    - 工序详情显示该逻辑模板实际覆盖的产品清单
  - 主列表恢复编辑入口，便于直接从模板组进入维护，不必只先点查看。

  - 生产工序管理统计卡

### 19. 应收收款与总账流水自动补齐，避免“已收款但总账为空”

- 目的：
  - 避免应收账款已经登记收款、状态也变成已收/已付清，但总账管理仍然查不到任何收款凭证。
  - 把“应收收款”和“总账流水”统一回同一条业务链，防止后续测试时出现口径不一致。
- 逻辑：
  - 以后只要应收记录存在已收金额，系统会自动确保存在对应的 `payment_records` 收款流水。
  - 如果是旧数据、导入数据或历史路径只改了应收没有生成流水，系统会在读取收付款记录时自动补建缺失流水。
  - 补建时优先使用应收单自己的收款账户、收款日期；如果缺失，则回退销售订单上的收款账户。
  - 补建流水只补“差额”，不会重复生成已存在的收款记录。
- 影响范围：
  - 应收收款更新
  - 总账管理
  - 收付款记录列表
- 影响文件：
  - `server/db.ts`

### 20. 总账管理按真实收付款流水映射显示，避免字段空白和金额为 0

- 目的：
  - 避免总账管理已经查到收付款流水，但页面仍把凭证号、日期、摘要、金额显示成空值或 `¥0`。
  - 让财务测试时看到的总账列表与实际 `payment_records` 数据一致。
- 逻辑：
  - 总账管理页读取 `payment_records` 后，统一先映射成凭证展示结构。
  - `recordNo` 映射为凭证号，`paymentDate` 映射为日期，`remark` 映射为摘要。
  - 对收付款流水，如果没有独立状态字段，默认按“已过账”展示。
  - 金额按流水金额映射到凭证总额，不再出现列表有记录但借贷金额都显示 `0` 的情况。
- 影响范围：
  - 总账管理列表
  - 总账统计卡
- 影响文件：
  - `client/src/pages/finance/Ledger.tsx`

### 12. 人员补列逻辑改为可重复执行，人员档案新增/修改后自动同步到用户管理

### 25. 到货管理删除规则统一为“未入库可删，已入库不可删”

- 目的：
  - 让采购到货单在到货管理里具备明确可测试的删除逻辑，不再只允许“待质检”删除，避免不同状态下操作口径不一致。
  - 同时防止已入库到货单被误删，造成库存与到货、IQC 记录断链。
- 逻辑：
  - 到货管理前端对真实到货单统一显示删除入口，只要状态不是 `warehoused` 就允许发起删除。
  - 后端删除接口增加正式校验：如果到货单状态已是 `warehoused`，或已经挂了入库单号 `inboundDocumentNo`，则拒绝删除。
  - 未入库到货单删除时，仍继续级联清理关联的 IQC 明细与到货明细，并同步回写采购订单进度。
- 影响范围：
  - 采购部 -> 到货管理
  - 到货单与 IQC 的联动链
  - 采购订单收货/到货进度
- 影响文件：
  - `client/src/pages/purchase/GoodsReceipt.tsx`
  - `server/db.ts`

- 用户明确提出“修改”时，默认按“修改系统逻辑/匹配规则/联动规则”理解，不按“只修当前一条数据”理解；除非用户明确说只是修数据。

- 每条日志至少包含：
  - 变更目的
  - 变更逻辑
  - 影响范围
  - 影响文件

## 2026-03-16 编码规则统一收口

- 编码规则页改成以系统托管规则为准，旧的重复规则行不再混入展示，避免同模块同时出现“真实流水号几百 / 规则页还是 0”的假冲突。

- 当前原则统一为：新建业务单据默认不手工拼号、不前端乱生号；有历史真实编号就保留，没有就由系统规则生成，方便后期排查与追溯。
